## 目的

このガイドは、屋外車椅子搬送ロボットシステムの完全なドキュメントセットを、異なるステークホルダーがナビゲートするのを支援します。各役割を、その責任に必要な特定のドキュメントにマッピングします。

### 1. プロジェクトマネージャー

- `REQUIREMENTS_TRACEABILITY_MATRIX.md` - 100%カバレッジを確認
- `PERFORMANCE_REQUIREMENTS.md` - 予算とパフォーマンス目標

📄 **タイムラインと計画（重要）**
- `PROJECT_TIMELINE_WITH_AI.md` - 完全なプロジェクトタイムライン（24週間、AI加速）

  - 並行作業の機会
  - リソース配分（1〜2名の開発者 + テスター）
  - リスク管理と緊急時対応計画
- `COMPLETE_TESTING_PLAN_WITH_TIMELINE.md` - 包括的テスト計画
  - 3層テスト戦略（単体 → 統合 → フィールド）

  - 376以上のテストケース（88% AI生成）
  - フィールド検証シナリオ

**いつ使用するか:**
- プロジェクトキックオフ: システム要件 + プロジェクトタイムラインを読む
- 週次ステータス: タイムラインマイルストーン + テスト進捗をチェック
- 予算レビュー: ハードウェア要件BOM + タイムラインリソース配分を参照
- ステークホルダー更新: パフォーマンス要件メトリクス + マイルストーン追跡を使用

**アクションアイテム:**
- [ ] 92のシステム要件すべてにサブシステム内訳があることを確認
- [ ] タイムラインマイルストーン経由で実装進捗を追跡（週次）
- [ ] ハードウェア要件BOMに対するコストを監視
- [ ] 受入基準に対するパフォーマンスを報告

📄 **プロジェクトタイムライン**
- `PROJECT_TIMELINE_WITH_AI.md`
  - 開発と統合されたテストタイムライン
  - リソース配分（1〜2名の開発者 + 専任テスター）

**テストワークフロー:**
1. **要件カバレッジ:** 各要件をテストケースにマッピング（合計376以上）
2. **単体テスト（AI加速）:** AIが80%以上のテストケースを生成、開発者がレビュー/改良
3. **統合テスト:** Gazeboシミュレーション + ROS 2 launchテストを使用
4. **フィールドテスト（段階的）:** 開発後に各サブシステムをテスト（サブシステムごとに0.5〜1.5週間）
5. **受入:** MVP基準に対して検証（>90%ミッション成功、第24週）
6. **自動化:** CI/CDパイプライン（GitHub Actions、自動テスト実行）

**主要メトリクス:**
- コードカバレッジ目標: 87%以上（ライン）、74%以上（ブランチ）
- 総テストケース: 376以上（88% AI生成）
- MVP成功基準: >90%ミッション成功率
- 本番成功基準: >95%ミッション成功率
- ドッキング精度: ±5mm（屋外）、±2mm（屋内目標）
- フィールドテスト期間: 約10週間（24週間プロジェクト全体で）

**参照セクション:**
- 単体テストの例: `COMPLETE_TESTING_GUIDE.md:30-150`

**オペレーターワークフロー:**

2. **システムチェック:** すべてのノードが実行中、ステータスが緑であることを確認
3. **ミッション選択:** UIまたは設定ファイルからルートをロード
4. **モニタリング:** ロボットステータス、安全性ステータス、ドッキングステータスを監視
5. **緊急対応:** E-Stop手順、リカバリーステップ

**重要なセクション:**
- 日常起動: `COMPLETE_DEPLOYMENT_GUIDE.md:320-360`
- ミッション実行: `COMPLETE_DEPLOYMENT_GUIDE.md:420-480`
- 緊急手順: `COMPLETE_DEPLOYMENT_GUIDE.md:520-600`

**クイックリファレンスカード:**
- E-Stop手順: 押す → 停止確認 → 評価 → リセット → 再開
- 自己位置推定喪失: 停止 → 手動再配置 → RVizで姿勢設定 → 再開
- バッテリー低下（<20%）: ミッション中止 → 充電器にナビゲート → 80%以上まで充電

## ドキュメント更新ワークフロー

### いつドキュメントを更新するか

**要件変更:**
1. 影響を受ける要件ドキュメントを更新
2. トレーサビリティマトリックスを更新
3. 必要に応じてアーキテクチャ/設計に伝播

**アーキテクチャ変更:**
1. アーキテクチャドキュメントを更新
2. 影響を受ける設計ドキュメントを更新
3. インターフェース仕様を更新
4. すべての開発者に通知

**インターフェース変更（重要）:**
1. `INTERFACE_SPECIFICATIONS_COMPLETE.md`を更新
2. トピックが変更された場合は`ROS2_TOPICS.md`を更新
3. すべてのチームに通知（破壊的変更！）
4. インターフェースを使用する設計ドキュメントを更新

**実装発見:**
1. 実際の実装で設計ドキュメントを更新
2. 一般的なパターンの場合は開発ガイドにメモを追加
